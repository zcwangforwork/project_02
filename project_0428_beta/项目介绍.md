# 医疗器械体系文件审核 Agent — 项目介绍文档

## 一、项目概述

**项目名称**: 医疗器械体系文件审核 Agent (Medical Device System Document Audit Agent)

**当前版本**: 3.0.0

**项目定位**: 基于 RAG（检索增强生成）技术的医疗器械企业体系文件智能审核系统。用户上传医疗器械质量管理体系文件（Word/PDF），系统结合预处理的法规知识库，通过大语言模型自动输出审核意见，包括缺失内容、修改建议和关联法规依据。

**目标用户**: 医疗器械企业质量管理人员、注册工程师、体系审核员

**核心能力**:
- 文档上传审核：支持 .docx / .pdf 文件上传
- RAG 智能检索：结合预处理的法规知识库进行相关标准检索
- 体系文件审核：指出缺失内容、需要修改的地方、具体修改建议
- 法规条款关联：自动关联 ISO 13485、ISO 14971、IEC 62304、EU MDR、FDA QSR、NMPA GMP 等标准条款
- 审核报告导出：支持将审核结果导出为 Word (.docx) 格式

### 3.3 数据流

1. 用户通过前端上传文档 → POST `/api/analyze`
2. `main.py` 接收文件，流式写入临时文件
3. `doc_processor.py` 解析文档为结构化 Markdown
4. `rag_retriever.py` 按 Markdown 标题分段（最多 20 段）
5. 每段并发（Semaphore=2）查询 ChromaDB 检索相关法规标准
6. 每段调用 LLM 生成审核意见
7. 汇总所有段审核结果，调用 LLM 生成综合报告
8. 返回 Markdown 格式审核报告，前端渲染展示
9. 可选：导出为 Word 文档

### 3.4 审核模式

系统支持两种审核模式：
- **risk_management**：聚焦 ISO 14971:2019 风险管理，使用专用审核 Prompt
- **general**：通用审核，覆盖 ISO 13485、ISO 14971、IEC 62304、EU MDR、NMPA GMP

## 四、代码文件详解

### 4.1 核心模块

#### `backend/main.py` (851 行) — FastAPI 主程序

项目入口和 API 服务核心，包含：

- **Config 类**：从环境变量读取 API URL、API Key、模型名称、超时时间等配置
- **ConversationHistory 类**：基于 session_id 的对话历史管理，支持截断（单会话最多 5 条消息，单条最大 8000 字符，总会话最大 50000 字符）
- **LLM 调用封装** (`call_openai_api()`)：3 次指数退避重试机制，处理限流、超时、连接错误
- **报告导出** (`_md_to_docx()`)：将审核报告 Markdown 转为格式化 Word 文档（微软雅黑字体，居中标题）
- **9 个 API 端点**（详见第五节）
- **应用生命周期管理**：启动时初始化向量库，关闭时释放资源
- **并发控制**：`limit_concurrency=20`，`timeout_keep_alive=10`

#### `backend/rag_retriever.py` (647 行) — RAG 多轮审核管道

系统的核心智能模块，实现了多轮审核管道：

- **RAGRetriever 类**：核心审核引擎
- **三个专用 System Prompt**：
  - `RISK_MANAGEMENT_SECTION_PROMPT`：风险管理审核专用提示词
  - `GENERAL_SECTION_PROMPT`：通用体系文件审核提示词
  - `SYNTHESIS_PROMPT`：汇总综合报告生成提示词
- **analyze_document() 方法**：
  - Phase 1：调用 `split_by_markdown_headers()` 分段
  - Phase 2：`asyncio.gather` + `Semaphore(2)` 并发审核每段
  - Phase 3：调用 LLM 汇总生成最终综合报告
- **内存保护机制**：ChromaDB 查询加锁串行化、手动 `del + gc.collect()`、`MAX_SECTIONS=20`、单条回复截断至 30000 字符
- **智能检索优先级**：优先检索风险管理类文档（关键词：风险、risk、14971、42062、9706），再补充其他来源
- **降级模式**：无法分段时自动降级为单次审核调用

#### `backend/doc_processor.py` (461 行) — 文档解析提取

负责将各种格式文档解析为结构化 Markdown：

- **extract_text_from_docx()**：O(n) 段落遍历（已从 O(n²) 优化），按 Word 样式名识别标题层级，表格转 Markdown，检测列表项
- **extract_text_from_pdf()**：基于 pdfplumber，通过字体大小比例启发式检测标题层级
- **extract_text_from_doc()**：三级降级策略——win32com COM 自动化 → olefile 提取 → 二进制扫描中文字符
- **split_by_markdown_headers()**：按 Markdown 标题（H1-H6）将文本拆分为段落
- **chunk_text()**：按 500 字符重叠分块，用于知识库入库

#### `backend/vector_store.py` (348 行) — ChromaDB 向量库封装

封装 ChromaDB 的持久化和查询：

- **MiniMaxEmbeddingFunction**：自定义 ChromaDB Embedding 函数，调用火山引擎 Embedding API，失败时降级为随机向量
- **SQLite PRAGMA 注入**：拦截所有 `sqlite3.connect()` 调用，注入内存优化参数（`cache_size=-4000`, `mmap_size=0`, `temp_store=FILE`）
- **VectorStore 类**：
  - 主 collection：`medical_device_kb`，HNSW 参数 M=4, construction_ef=50, search_ef=30
  - 查询 collection：`["medical_device_kb_v2"]`（避免加载旧版 2.8GB HNSW 索引）
  - `add_documents()`：自动批处理（500 条/批）
  - `query()`：多 collection 合并，距离排序

## 六、知识库数据

### 6.1 ChromaDB 存储

- **存储位置**: `backend/data/chroma_db/`
- **主数据库**: `chroma.sqlite3` (~2.9 GB)
- **索引类型**: HNSW（分层可导航小世界图）
- **索引参数**: M=4, construction_ef=50, search_ef=30
- **索引段**: 3 个 HNSW 段文件夹，最大单段 747 MB

### 6.2 Collection 策略

- **`medical_device_kb`** (原版)：2.9 GB，历史数据保留，只读查询
- **`medical_device_kb_v2`** (新版)：当前活跃查询目标，避免加载大索引

### 6.3 知识库内容

源文档位于 `develop_documents/` 目录，包含医疗器械质量体系标准文件：
- ISO 13485:2016 质量管理体系
- ISO 14971:2019 风险管理
- IEC 62304 软件生命周期
- EU MDR 2017/745 欧盟医疗器械法规
- FDA 21 CFR Part 820 质量体系法规
- NMPA 医疗器械生产质量管理规范
